## Мнения из сообщества

### @claude_md_user1

> До CLAUDE.md: каждую сессию копировал описание проекта в первое сообщение. После: открыл папку — агент уже знает стек, правила, архитектуру. Сразу к делу.

Инвестиция 15 минут в CLAUDE.md окупается с первой же сессии.

### @three_levels

> CLAUDE.md работает на трёх уровнях: глобальный (`~/.claude/CLAUDE.md`) для всех проектов, проектный (`./CLAUDE.md`) для конкретного проекта, модульные (`.claude/rules/*.md`) для отдельных доменов. Агент объединяет все три.

Три уровня — три разных типа правил. Не путать.

### @what_to_write

> Что писать в CLAUDE.md: стек и версии, архитектурные решения и почему они такие, что НЕ делать, известные баги и их решения. Последнее особенно ценно — агент не наступает на грабли, на которые уже наступали.

Раздел «Чего НЕ делать» часто важнее «Что делать».

### @tokens_saver

> Проверил: с хорошим CLAUDE.md первые ответы точнее и используют правильный стек. Агент не тратит 2-3 шага на «давай разберёмся что за проект». Токены идут на задачу, а не на разведку.

### @team_user

> В командных проектах CLAUDE.md — это онбординг для агентов. Новый разработчик подключает Claude Code → агент сразу работает по правилам команды. Как README, но для AI.

## Итог

**Минимальный CLAUDE.md:**

```markdown
# My Project

## Стек
- Python 3.12 + python-telegram-bot 20.x
- PostgreSQL на Railway
- Деплой: Railway (worker, не web)

## Правила
- async/await везде, не sync
- Type hints обязательны
- Коммиты на английском: feat:, fix:, docs:

## Известные проблемы
- Railway требует Procfile с `worker:` для ботов
- При обновлении БД — сначала миграция, потом деплой

## Чего НЕ делать
- Не трогать config/production.yml напрямую
- Не коммитить .env файлы
```

**Три уровня:**
- `~/.claude/CLAUDE.md` — глобальные правила для всех проектов
- `./CLAUDE.md` — правила конкретного проекта
- `.claude/rules/*.md` — модульные правила по доменам (api, database, deploy)

**Что писать:** стек + версии, архитектурные решения, что НЕ делать, известные баги.
