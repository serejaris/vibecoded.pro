## Мнения из сообщества

### @hooks_fan

> Правила в CLAUDE.md — вероятностные. Пишешь «всегда запускай линтер» — агент иногда забывает. Хук — детерминированный: запустится всегда, без исключений.

Хуки решают проблему «агент забыл» раз и навсегда.

### @dev_workflow

> Лучший хук — фиксация прогресса в конце сессии. Агент автоматически комментирует GitHub Issue с тем, что сделал. Между сессиями не теряешь контекст.

```json
{
  "hooks": {
    "PostSession": [{
      "command": "gh issue comment $CURRENT_ISSUE --body \"Итоги: $SESSION_SUMMARY\""
    }]
  }
}
```

### @security_dev

> Добавил хук безопасности: перед коммитом проверяет, не попали ли API-ключи в staged изменения. Агент физически не сможет закоммитить секрет.

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "git_commit",
      "command": "git diff --staged | grep -qE '(API_KEY|SECRET)' && exit 1 || exit 0"
    }]
  }
}
```

### @linter_user

> PostToolUse с ESLint — каждый файл автоматически линтится после записи. Не нужно помнить запускать линтер, не нужно просить агента. Просто работает.

### @context_keeper

> Главный юзкейс для меня: хук после сессии пишет в CLAUDE.md что было сделано и что осталось. Следующая сессия начинается с актуальным контекстом.

Решает главную проблему длинных проектов — потерю контекста между сессиями.

## Итог

Хуки настраиваются в `.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "write_to_file",
      "command": "npx eslint --fix $FILE"
    }],
    "PostSession": [{
      "command": "echo 'Session done' >> session.log"
    }]
  }
}
```

**Когда использовать хуки (а не CLAUDE.md):**
- Линтинг и форматирование кода
- Проверка безопасности перед коммитом
- Фиксация прогресса в Issues
- Автоматические тесты после изменений

**Правило:** критичное и обязательное — в хуки, гибкое и контекстуальное — в CLAUDE.md.
