## Мнения из сообщества

### @newcomer1

> Я не читаю код — тестирую как пользователь. Запускаю приложение, пробую все сценарии которые описал в задаче, потом пробую сломать: пустые поля, длинный текст, повторные клики. Если всё работает — ок.

Тестирование через UX, без чтения кода.

### @vibecoder1

> Всегда прошу агента: "После написания кода — напиши тесты и запусти их. Покажи результат." Если тесты падают — продолжаем. Тесты прошли — можно деплоить.

Автоматические тесты как часть промпта.

### @developer1

> Главное правило: не деплоить без проверки на локале. Даже если агент говорит "всё работает" — запускай сам, проверяй основные сценарии. AI уверенно говорит что работает, даже когда не работает.

Скептицизм как практика: агент может ошибаться и не знать об этом.

### @vibecoder2

> Золотое правило тестирования с AI: проверяй не только happy path (всё правильно), но и edge cases. Что если пользователь не авторизован? Что если база пустая? Что если запрос упал?

Edge cases важнее основного сценария.

## Код

```bash
# Попроси агента написать и запустить тесты
claude "Напиши тесты для функции createUser:
- успешное создание
- дублирующийся email (должна быть ошибка)
- пустое имя (должна быть ошибка)
Запусти тесты и покажи результат."

# Попроси агента самому проверить
claude "Запусти приложение, зайди на /dashboard, проверь что данные загружаются. Если ошибки — исправь."
```

```markdown
# Чеклист перед деплоем

## Функциональность
- [ ] Основной сценарий работает
- [ ] Форма валидируется (пустые поля, неверный формат)
- [ ] Ошибки показываются пользователю
- [ ] Авторизация работает

## Технически
- [ ] `npm run build` без ошибок
- [ ] `npm run test` все зелёные
- [ ] Нет console.error в браузере
- [ ] Работает на мобильном экране
```

## Итог

Тестирование с AI — это командная работа:
1. **Ты** — тестируешь как пользователь, проверяешь edge cases
2. **Агент** — пишет автоматические тесты, запускает их, исправляет баги
3. **Вместе** — не деплоите пока не прошло оба уровня проверки
