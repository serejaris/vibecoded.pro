## Мнения из сообщества

### @context_expert

> CLAUDE.md — главный инструмент управления контекстом. Агент читает его при каждом запуске, значит не тратит токены на разведку проекта заново. Хорошо написанный CLAUDE.md экономит 20-30% контекста.

Инвестиция в CLAUDE.md окупается с каждой сессией.

### @modular_rules

> Для больших проектов разбил правила на домены:
> ```
> .claude/rules/
>   ├── api.md      — правила для API
>   ├── database.md — миграции, RLS
>   └── deploy.md   — Railway, CI/CD
> ```
> Агент подгружает только то, что нужно для задачи.

Модульные правила — следующий уровень после базового CLAUDE.md.

### @new_chat_user

> Правило: когда качество ответов падает — не пытаюсь «починить» текущую сессию. Просто открываю новый чат и кратко описываю контекст. 30 секунд на формулировку лучше, чем работа в переполненном окне.

Чистый контекст > попытки реанимировать старый.

### @github_issues_user

> GitHub Issues как persistent storage — главный инсайт. Агент заканчивает сессию → хук фиксирует прогресс в Issue → следующая сессия начинается с чтения Issue. Контекст не теряется между сессиями.

### @mcp_limit

> Два MCP-сервера максимум — не просто рекомендация. Проверил на практике: три сервера, и 10% контекста уходит только на их описания. Агент заметно хуже понимает код.

## Итог

**Иерархия контекста:**

```
~/.claude/CLAUDE.md          — глобальные правила
    ↓
./CLAUDE.md                  — правила проекта
    ↓
.claude/rules/*.md           — модульные правила
    ↓
Текущий промпт
```

**Стратегии:**
1. **CLAUDE.md** — записывай стек, архитектуру, паттерны. Агент не тратит токены на разведку
2. **Новый чат** — когда качество падает, начни заново с кратким контекстом
3. **Максимум 2 MCP** — каждый сервер ест контекст
4. **GitHub Issues** — фиксируй прогресс между сессиями через хуки
5. **Планирование отдельно** — PRD и архитектурные решения лучше обсуждать в обычном чат-боте, а не в агенте

**Признак переполненного контекста:** агент повторяется, забывает решения, предлагает то, что уже сделано.
